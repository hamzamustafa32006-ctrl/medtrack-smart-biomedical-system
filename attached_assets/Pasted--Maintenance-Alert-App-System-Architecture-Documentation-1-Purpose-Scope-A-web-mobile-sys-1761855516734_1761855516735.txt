# Maintenance Alert App — System Architecture Documentation

## 1) Purpose & Scope

A web/mobile system that tracks biomedical/industrial equipment, maintenance schedules and contracts, then delivers timely alerts and supports basic work-order flows. Designed for single hospital or multi-facility (multi-tenant-ready).

---

## 2) High-Level Architecture

**Frontend (Web + PWA)**

* React (Vite/Next.js), TypeScript, Tailwind.
* Auth via OAuth/OIDC (e.g., Azure AD/Google) + JWT.
* Offline cache using Service Worker + IndexedDB for draft work orders.

**Backend**

* REST + Webhooks; optional GraphQL gateway.
* Node.js (NestJS/Express) or Python (FastAPI) with layered architecture (controllers → services → repositories).
* Domain services: Equipment, Schedule Engine, Alerts/Notifications, Contracts, Work Orders, Reports, Import/Export.

**Data Layer**

* Primary DB: PostgreSQL.
* Search & filtering: Postgres GIN/Trigram indexes; optional OpenSearch for full-text.
* Cache: Redis (sessions, rate limits, computed next-due dates).
* Object storage: S3/GCS for manuals, photos, certificates.

**Async & Scheduling**

* Message bus/queues: RabbitMQ/SQS.
* Job runner: BullMQ/Celery; cron-like schedules for daily alert computation.
* Real-time: WebSocket/SSE for in-app alerts.

**Integrations**

* Email (SES/SendGrid), SMS/WhatsApp (Twilio), Push (FCM/APNs), Calendar (ICS export), SSO, CMMS/ERP (outbound webhooks), Barcode/QR.

**Deployment**

* Containers (Docker) on AWS ECS/Fargate or GKE; IaC with Terraform.
* CDN + WAF (CloudFront).
* GitHub Actions CI/CD with blue-green or canary.

---

## 3) Core Requirements

* Track equipment, contracts, locations, vendors, maintenance plans, and history.
* Compute next-due tasks and send multi-channel alerts with snooze/escalation.
* Role-based access (RBAC) with audit logs.
* Import/export (CSV/Excel/JSON).
* Reports (compliance, overdue, MTBF/MTTR basics).
* Multi-facility and optional multi-tenant isolation.

## 6) Scheduling & Alert Logic

**Computation**

1. For each active `maintenance_plans` record, compute `next_due = last_event_date + frequency_days`.
2. Apply `buffer_days` for early warnings.
3. Generate/refresh an **Open** `maintenance_task` if none exists within window.
4. Alert policy:

   * T-buffer: Info alert to assignee + supervisor.
   * T-0: Warning alert (email + push + in-app).
   * T+X overdue: escalate levels (e.g., 3, 7, 14 days) → manager → biomedical head.

**Retry & De-dup**

* Redis keys `alert:{plan_id}:{due_date}` with TTL to prevent duplicates.
* Exponential backoff for provider failures.

**Example Cron Jobs**

* `0 3 * * *` Generate tasks/alerts for next 30 days.
* `*/10 * * * *` Notification dispatcher.
* `0 * * * *` SLA breach scanner for work orders.

## 7) Security

* OIDC + short-lived access tokens, refresh rotation.
* RBAC: roles → permissions (JSON). Examples:

  * **Technician:** read equipment, complete tasks.
  * **Supervisor:** assign tasks, close WOs, export.
  * **Admin:** user mgmt, tenant config.
* Row-level security by `tenant_id`.
* Secrets in AWS Secrets Manager; KMS encryption.
* S3 buckets private + signed URLs; server-side encryption.
* Audit logging for all write ops; immutable storage option.
* Rate limiting (per IP + per user) via Redis.
* Input validation & content scanning for uploads.
* GDPR-style data export/delete (per tenant).

## 9) Data Lifecycle, Backups & DR

* PITR for Postgres; daily full + 7-day WAL retention.
* S3 versioning + lifecycle (IA → Glacier after 90 days).

## 11) Example Sequences (Textual)

**Create Equipment & Plan**

1. User submits form → `POST /equipment`.
2. Backend stores record, emits `equipment.created` event.
3. User adds plan → `POST /plans`.
4. Scheduler picks plan tonight → generates first `maintenance_task` if due window falls within horizon.
5. Notification service sends T-buffer alert.

**Complete Task**

1. Tech opens task → uploads results → `POST /tasks/{id}/complete`.
2. Service validates checklist → closes task → writes `maintenance_events`.
3. Recomputes `next_due` → creates next task if necessary → may cancel open alert.
